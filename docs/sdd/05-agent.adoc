= Agent Pipeline Design

== Overview

The agent module implements the University Chair and its sub-agents as stateful Go
structs wrapping Anthropic SDK conversations. The Chair owns all student-facing
interaction: intake, syllabus generation, due-date assignment, and active-course
chat. Post-approval content generation is delegated to Professor agents (one per
lesson section) that each receive a `web_search` tool and are reviewed by a
Reviewer agent before content is committed. All agent activity is observable
through the `pipeline_events` table, streamed to the frontend via SSE.

CREATE TYPE agent_run_type AS ENUM (
    'intake',
    'syllabus',

    id              UUID             PRIMARY KEY DEFAULT gen_random_uuid(),
    course_id       UUID             NOT NULL REFERENCES courses(id) ON DELETE RESTRICT,

CREATE INDEX agent_runs_course_id_idx ON agent_runs (course_id);

=== Migration Plan

Sequential migration files in `backend/agent/migrations/`. ENUM values are
added with `ALTER TYPE ... ADD VALUE`. No ENUM values are removed without a
multi-step migration that first migrates all rows off the removed value.
`pipeline_events.payload` is JSONB so new event fields are additive and
require no schema change.

== Agent Architecture

// @{"req": ["REQ-AGENT-015", "REQ-SYS-021"]}

The `UniversityChair` is a Go struct holding a slice of Anthropic
`MessageParam` values (the conversation history) and a reference to the
Anthropic SDK client. Conversations are scoped by `course_id`. On each call to
`chair.Send(ctx, studentMessage)`:

[source,go]
----
type UniversityChair struct {
    client   anthropic.Client
    courseID uuid.UUID
    history  []anthropic.MessageParam
    db       *pgxpool.Pool
}
----

Each student message is appended to `history`, sent to the Claude API, and the
response is appended and persisted to `chat_messages` in the same transaction.
The Chair reuses the same conversation context across intake, syllabus drafting,
due-date assignment, and active-course chat. Context switching between phases is
done by injecting system-prompt segments at the start of each phase, not by
creating a new conversation.

After the student approves the syllabus, the Chair agent summarises the full
intake conversation into a compact system-prompt segment. The summary is a
structured record of: knowledge level, learning objectives, time availability,
and any special constraints noted during intake. The raw `chat_messages` rows
from the intake phase are retained in the database for audit, but the in-memory
`[]anthropic.MessageParam` slice is replaced with a single summarised system
message before the syllabus-drafting phase begins. This bounds token cost for
long-running courses without losing the context required for active-course chat.

Professor agents are short-lived: one `ProfessorAgent` struct per lesson section,
instantiated at generation time and discarded after the section is committed.

== Intake Questionnaire Flow

// @{"req": ["REQ-AGENT-001", "REQ-AGENT-002", "REQ-AGENT-009", "REQ-SYS-009", "REQ-SYS-010", "REQ-SYS-031"]}

On course creation (`status = intake`), the Chair sends the first intake question
as an `assistant` message. The intake sequence collects, at minimum:

. Current knowledge level (self-assessed 1-5 scale or free text)
. Specific learning objectives (at least one)
. Weekly time availability in hours

The Chair tracks a `intakeFieldsSatisfied` bitmask in memory (not persisted) and
continues asking follow-up questions until all three fields are satisfied. Minimum
threshold is enforced in the Chair's `isIntakeSufficient()` method; it does not
call the syllabus-generation tool until the method returns true.

Once intake is complete, the Chair:

. Generates a syllabus draft via a structured tool call that returns an AsciiDoc
  document (max 500 lines per the content composition rule).
. Writes the syllabus to `syllabi` with `version = 1`, `approved_at = NULL`.
. Transitions the course to `syllabus_draft`.
. Emits a `syllabus_draft_ready` pipeline event.

Grade distribution is embedded in the syllabus AsciiDoc by the Chair tool call
at this stage (REQ-AGENT-002). The Chair derives point weights from topic complexity
and the number of sections; no fixed distribution is hardcoded.

== Content Library Reuse Consent

// @{"req": ["REQ-AGENT-004", "REQ-SYS-012"]}

Before generation begins, the Go `contentLibrary.FindMatching(topic, sections)`
query searches existing lesson content for cosine-similar topics (using
`pg_trgm` trigram similarity as a proxy until a vector store is integrated).
If matches are found, the Chair presents them to the student in chat:

[source,json]
----
{
  "event_type": "section_generating",
  "payload": {
    "section_id": "...",
    "reuse_candidate": true,
    "candidate_title": "Introduction to Paxos"
  }
}
----

Generation for that section does not proceed until the student sends an explicit
consent message, captured as a `chat_messages` row and parsed by the Chair. If
consent is denied, a new section is generated from scratch.

== Content Generation Pipeline

// @{"req": ["REQ-AGENT-003", "REQ-AGENT-005", "REQ-AGENT-006", "REQ-AGENT-007", "REQ-AGENT-008", "REQ-AGENT-010", "REQ-SYS-011", "REQ-SYS-014", "REQ-SYS-015", "REQ-SYS-016", "REQ-SYS-034"]}

On `syllabus_approved`, the course module publishes an internal Go channel event.
The agent runner subscribes and:

. Creates an `agent_runs` row with `run_type = 'content_generation'`.
. Emits `generation_started`.
. Iterates over each syllabus section:
.. Instantiates a `ProfessorAgent` with a `web_search` tool.
.. Emits `section_generating` with section metadata.
.. Professor calls Claude; tool calls to `web_search` are executed synchronously
   by the Go runner and results injected back into the conversation.
.. Completed section AsciiDoc is handed to a `ReviewerAgent`.
.. Reviewer checks for at least one citation; if absent, emits
   `section_review_failed` and increments `agent_runs.iteration_count`.
.. If `iteration_count >= max_iterations` (from admin config table), emits
   `correction_escalated`, sets `agent_runs.status = 'failed'`, and halts.
.. On review pass, emits `section_review_passed`, persists the lesson content.
. On all sections committed, transitions course to `active`, emits
  `generation_complete`, sets `agent_runs.status = 'completed'`.

Section regeneration (`POST /api/v1/courses/:id/sections/:section_id/feedback`)
creates a new `agent_runs` row with `run_type = 'section_regen'` and runs the
same Professor + Reviewer loop for the single section, replacing the stored
content on success.

== Correction Loop

// @{"req": ["REQ-AGENT-007", "REQ-AGENT-008", "REQ-SYS-016"]}

`agent_runs.iteration_count` is incremented atomically with an
`UPDATE agent_runs SET iteration_count = iteration_count + 1 WHERE id = $1`
before each correction attempt. The max-iterations value is read from the
`system_config` table key `correction_loop_max_iterations` at runner start; it
is not re-read mid-run to avoid mid-run policy changes causing inconsistency.

The default value for `correction_loop_max_iterations` in `system_config` is 3.
The migration that seeds `system_config` includes this default as an `INSERT`
with `ON CONFLICT DO NOTHING` so that existing deployments with a custom value
are not overwritten.

When the cap is reached:

. `agent_runs.status` is set to `failed`.
. A `correction_escalated` pipeline event is emitted with `payload.section_id`.
. The notify module is called to deliver an admin alert.
. The student receives a `api_failure` pipeline event via SSE.

== Exponential Backoff

// @{"req": ["REQ-AGENT-012", "REQ-SYS-044"]}

The Anthropic SDK wrapper `anthropicClient.Do(ctx, req)` inspects the HTTP
response status. On 429:

[source,go]
----
delay := min(basDelay * (1 << attempt), maxDelay)
jitter := time.Duration(rand.Int63n(int64(delay / 4)))
time.Sleep(delay + jitter)
----

`base_delay` defaults to 1 second; `max_delay` defaults to 60 seconds. Both are
configurable via `admin_config`. After exhausting `max_retries` (also configurable),
the wrapper returns a sentinel `ErrRateLimitExhausted` error, which the runner
maps to an `api_failure` event and halts generation per REQ-AGENT-011.

== Internet Search

// @{"req": ["REQ-AGENT-005", "REQ-SYS-014"]}

The `web_search` tool is registered with each `ProfessorAgent` as an Anthropic
tool definition. When Claude emits a `tool_use` block with `name = "web_search"`,
the Go runner:

. Extracts the `query` field from the tool input.
. Calls the Brave Search API (configured via `BRAVE_API_KEY` environment variable).
. Returns the top-5 results as a `tool_result` message.
. Appends both the tool call and result to the conversation before the next API
  call.

The search API key is supplied exclusively via environment variable, consistent
with the credential management pattern in REQ-SYS-007.

== Pipeline Status and SSE

// @{"req": ["REQ-AGENT-006", "REQ-SYS-015"]}

`GET /api/v1/courses/:id/pipeline/events` opens a chunked HTTP response with
`Content-Type: text/event-stream`. The handler polls `pipeline_events` for rows
with `emitted_at > last_seen_id` every 500 ms using long-poll semantics. The
500 ms poll interval is the confirmed target; it provides adequate responsiveness
for pipeline progress updates without requiring a LISTEN/NOTIFY infrastructure.
Each row is serialised as:

----
id: <pipeline_event.id>
event: <event_type>
data: <payload JSON>

----

The frontend subscribes with the EventSource API. On reconnect, the `Last-Event-ID`
header is used to resume from the last delivered event. Events are never deleted
from `pipeline_events`; the full history is available for audit and replay.

== Safe Termination Before Account Deletion

// @{"req": ["REQ-AGENT-013", "REQ-SYS-045"]}

Account deletion calls `agentRunner.TerminateForStudent(ctx, studentID)` before
any data deletion executes. This function:

. Sets all `running` `agent_runs` for the student to `status = terminated` in
  a single UPDATE.
. Cancels the associated Go `context.Context` values held in an in-memory
  `runRegistry` map keyed by `agent_run_id`.
. Waits up to 5 seconds for goroutines to acknowledge cancellation via a
  WaitGroup.

Agent goroutines call `ctx.Err()` between every API call and return immediately
on cancellation. No partial lesson content is committed after the context is
cancelled because content writes are guarded by `ctx`.

== Timeout Detection

// @{"req": ["REQ-AGENT-014", "REQ-SYS-043"]}

A background goroutine in the agent runner polls every 60 seconds:

[source,sql]
----
SELECT id, course_id FROM agent_runs
WHERE status = 'running'
  AND run_type = 'content_generation'
  AND started_at < now() - ($1 * INTERVAL '1 second');
----

`$1` is the `content_generation_timeout_seconds` value from `system_config`.
On match, the goroutine emits a `generation_timeout` pipeline event with
`payload.agent_run_id`. The notify module listens for this event type and
delivers a student-facing notification per REQ-NOTIFY-002 / REQ-SYS-043.
The run itself is not forcibly terminated by timeout detection alone; termination
requires an explicit admin action or account deletion.

== API Contract

// @{"req": ["REQ-AGENT-001", "REQ-AGENT-006", "REQ-AGENT-015", "REQ-SYS-009", "REQ-SYS-015", "REQ-SYS-021"]}
=== POST /api/v1/courses/:id/chat

Append a student message and receive the Chair's response. Persists both roles
to `chat_messages`.

Request:
[source,json]
----
{ "message": "I want to focus on Raft, not Paxos." }
----

Response 200:
[source,json]
----
{
  "message_id": "m1a2b3c4-...",
  "role": "assistant",
  "content": "Understood — I'll adjust the syllabus to emphasise Raft...",
  "created_at": "2026-04-25T10:00:00Z"
}
----

Response 409 (`AGENT_RUN_TERMINATED`):
[source,json]
----
{ "error": "AGENT_RUN_TERMINATED", "message": "no active agent run for this course" }
----

=== GET /api/v1/courses/:id/chat/history

Returns paginated chat history for the course, ordered ascending by `created_at`.

Query params: `limit` (default 50), `cursor` (opaque, based on `created_at` + `id`).

Response 200:
[source,json]
----
{
  "messages": [
    { "id": "m1a2b3c4-...", "role": "student",    "content": "...", "created_at": "..." },
    { "id": "m1a2b3c5-...", "role": "assistant",  "content": "...", "created_at": "..." }
  ],
  "next_cursor": "eyJjcmVhdGVkX2F0..."
}
----

=== GET /api/v1/courses/:id/pipeline/events (SSE)

Streams `pipeline_events` rows as Server-Sent Events. Resumable via
`Last-Event-ID` header.

Response headers:
----
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
----

Example event stream:
----
id: e1f2a3b4-...
event: generation_started
data: {"agent_run_id":"r1a2b3c4-...","section_count":6}

id: e1f2a3b5-...
event: section_generating
data: {"section_id":"s1a2b3c4-...","title":"Introduction to Raft"}

id: e1f2a3b6-...
event: section_review_passed
data: {"section_id":"s1a2b3c4-...","iteration_count":1}
----

== Alternatives Considered

* *Separate conversation context per lifecycle phase* — rejected because
  partitioning the conversation loses prior context that the Chair uses to
  personalise later phases. A single growing context with phase-injected system
  segments preserves continuity at acceptable token cost.

* *Webhook delivery for pipeline events instead of SSE* — rejected because
  webhooks require student-owned receiving endpoints. SSE from the backend is
  simpler for browser clients and eliminates the need for ngrok-style tunnelling
  in development.

* *Vector similarity for content library matching* — deferred to a future
  milestone. `pg_trgm` trigram similarity is sufficient for the initial content
  library which is expected to be small. The data model does not preclude adding
  a `pgvector` column later.
